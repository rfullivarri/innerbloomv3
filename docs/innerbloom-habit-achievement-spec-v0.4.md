# Innerbloom — Habit Achievement Feature Spec v0.3
_Last updated: 2026-03-31_

## 1. Purpose

Habit Achievement is the third adaptive layer missing from Innerbloom.

Today the product already has:
- monthly task difficulty recalibration
- mode / rhythm upgrade suggestion
- persisted performance history per task

What is missing is the system that says:

**this task is no longer just “in progress”; it has become a real achieved habit.**

Habit Achievement does not replace Growth Calibration.
Habit Achievement does not replace Mode Upgrade Suggestion.

It sits on top of them.

### Final adaptive stack
1. **Growth Calibration**
   - monthly task difficulty adjustment
2. **Mode Upgrade Suggestion**
   - user-level suggestion to move to the next rhythm / game mode
3. **Habit Achievement**
   - task-level detection of sustained consolidation over time

This is the product layer that finally turns “lasting habits” into real product behavior.

---

## 2. Source of truth

Habit Achievement must reuse the persisted monthly data already generated by Growth Calibration.

### Primary source
`task_difficulty_recalibrations`

Use only rows with:
- `source = 'cron'`

Relevant fields:
- `task_id`
- `user_id`
- `period_start`
- `period_end`
- `expected_target`
- `completions_done`
- `completion_rate`
- `action`
- `game_mode_id`

### Why
This avoids duplicated logic and keeps Habit Achievement aligned with the same adaptive performance engine already used by Innerbloom.

---

## 3. Monthly execution order

Habit Achievement is evaluated monthly, after the monthly Growth Calibration flow is completed.

### Execution order
1. monthly task difficulty recalibration runs
2. recalibration rows are persisted
3. monthly mode-upgrade aggregation runs
4. habit achievement evaluation runs

Pipeline:

**recalibrate -> aggregate -> detect achieved habits**

---

## 4. Mathematical rule — approved v1

A task qualifies as a **Habit Achievement candidate** when all of the following are true.

### Rule 1 — minimum window
The task has **3 consecutive monthly cron periods** available.

### Rule 2 — weighted aggregate score
Main score:

`achievement_rate_3m = SUM(completions_done) / SUM(expected_target)`

across the latest 3 valid monthly periods.

### Rule 3 — threshold
`achievement_rate_3m >= 0.80`

### Rule 4 — monthly consistency guardrail
At least **2 of the 3 months** must have:

`completion_rate >= 0.80`

### Rule 5 — floor guardrail
None of those 3 months may fall below a minimum floor.
Initial default:

`completion_rate >= 0.50`

### Important implementation rule
The following values must be configurable:
- aggregate threshold (`0.80`)
- minimum months meeting goal (`2 of 3`)
- floor threshold (`0.50`)
- window size (`3 months`)

Do not hardcode these as permanent product constants.

## 5. Lifecycle states

Habit Achievement requires explicit lifecycle states beyond the current generic `active` flag.

## Final states

### A. `active`
Normal task.
- appears in Daily Quest
- appears in active task lists
- can appear in streak-related surfaces
- participates in Growth Calibration
- participates in Mode Upgrade analysis
- participates in Habit Achievement analysis

### B. `achievement_pending`
The backend has determined that the task qualifies as an achieved habit, but the user has not yet made a decision.

Final behavior:
- does **not** appear in Daily Quest
- does **not** appear in active task lists
- does **not** participate in Growth Calibration
- does **not** participate in Mode Upgrade analysis
- does **not** participate in new Habit Achievement evaluation
- stays in stand-by until user decision or until pending expires
- is only surfaced through the pending-achievement entry points

This is a frozen temporary state.

### C. `achievement_maintained`
The user chooses to keep tracking the achieved habit.

Final behavior:
- appears again in Daily Quest
- appears again in active task lists
- can be marked daily
- does **not** appear in Top Streaks logic
- does **not** generate task streak logic as an achieved-habit streak system
- shows achievement seal
- difficulty is fixed to lowest level
- generates only **1 Growth Point**
- excluded from Growth Calibration
- excluded from Mode Upgrade analysis
- excluded from future Habit Achievement evaluation
- also appears in the rewards shelf

### D. `achievement_stored`
The user chooses to archive / store the achieved habit.

Final behavior:
- removed from Daily Quest
- removed from active task lists
- removed from streak surfaces
- does not generate points
- excluded from all adaptive analyses
- only appears in the rewards shelf

---

## 6. Pending expiration rule

`achievement_pending` is not indefinite.

### Final rule
A task can stay in `achievement_pending` for **10 days**.

If the user does not decide within that time:
- the pending state expires
- the task returns to the normal analysis flow
- the oldest month in the original 3-month window falls out naturally
- future achievement evaluation uses the rolling current 3-month window again

This means the task can become achievement-pending again later if it still qualifies under the new rolling window.

---

## 7. User decision flow

When a task becomes `achievement_pending`, the user must choose between two outcomes:

### Option 1 — Maintain
The user continues tracking the habit as an already-achieved habit.

Final result:
- returns to Daily Quest
- returns to active task surfaces
- shows achievement seal
- generates only 1 GP
- no longer recalibrates
- no longer counts for mode upgrade
- no longer counts for achievement analysis
- also lives in the shelf

### Option 2 — Store
The user archives the habit as already achieved.

Final result:
- leaves active tracking
- stops generating points
- no longer appears in active surfaces
- remains permanently visible in the shelf

---

## 8. Re-activation from shelf

A stored achieved habit can later be turned back into a maintained achieved habit.

### Mechanism
Inside the shelf detail / medal back face there is a toggle, visually like an Apple-style switch.

- Toggle ON = maintained
- Toggle OFF = stored

### Final behavior
If a stored habit is turned ON:
- it becomes `achievement_maintained`
- it returns **immediately** to Daily Quest
- it returns **immediately** to active panels / active surfaces
- it keeps achievement status
- it remains excluded from adaptive engines
- it remains fixed at 1 GP

If a maintained habit is turned OFF:
- it becomes `achievement_stored`
- it disappears again from active surfaces
- it remains in the shelf

This toggle is the single state-switch mechanism between maintained and stored.

---

## 9. Irreversibility of achievement

Once a task has been officially marked as achieved:
- the achievement is never deleted
- the user may switch between maintained and stored
- but the achieved-habit history remains permanent

There is no “un-achievement” in v1.

---

## 10. Editing tasks

## 10.1 Edit inside the first 5 weeks
If the user edits a task **within the first 5 weeks** of that task’s life:
- it remains the same task
- the historical continuity is preserved
- it can still count toward the 3-month achievement window

### Final rule
Inside the first 5 weeks, **all edits preserve continuity**.
Do not try to distinguish cosmetic vs structural edits in v1.

This is the adjustment period.

## 10.2 Edit from week 6 onward
If the user edits the task **from week 6 onward**:
- the old task is treated as finished
- the old task becomes inactive
- the old task leaves active surfaces
- all previously earned GP remain preserved
- the edited version becomes a **new task**
- the new task starts from zero for:
  - Growth Calibration lifecycle
  - Habit Achievement window
  - future achievement analysis

### Technical direction
From UX, it should feel like a normal edit.

From backend, the preferred strategy is:

**clone new task + deactivate old task**

That matches the desired product behavior best.

## 10.3 Edit after achievement
If an achieved task is edited later:
- treat it as a new task lifecycle
- keep the old achieved version in historical records / shelf
- the edited version starts from zero as a new task

## 12. GP / XP definition in the medal detail

In this feature, XP and GP are treated as the same thing for display purposes.

### Final definition
The medal detail shows:

**total historical GP generated by that task up to the moment it became achieved**

This is the main historical number.

### Maintained follow-up counter
If the habit is maintained after achievement, show an additional small secondary counter indicating the extra GP earned since achievement.

Example concept:
- main number: historical GP before achievement
- small secondary `+N` number: GP added through maintained follow-up since achievement

This secondary number is optional in final UI polish, but the product intent is approved.

---

## 13. Streak behavior

Achieved habits do **not** participate in the Top Streaks concept as standard streak competitors.

### Final behavior
For `achievement_maintained`:
- task may still appear in active lists
- task may still visually show daily completion behavior
- but it does **not** enter Top Streaks ranking logic
- there is no special achieved-habit streak leaderboard concept in v1

For `achievement_stored`:
- no streak presence at all

---

## 14. Visual system

## 14.1 Achievement seal
Each achieved habit uses a seal / medal concept.

### Shape direction
- hexagon or medal-like badge

### Identity
- one standardized seal per trait
- visually organized by pillar
- total concept space: 30 seals (10 traits × 3 pillars)

### Front face
- seal visual
- task name underneath

### Back face
Keep it minimal.
Only include:
- achieved date
- total historical GP generated by that task up to achievement
- maintain toggle
- optionally one very small extra helper datum if needed later

Do not add more content in v1.

---

## 14.2 Shelf behavior
The rewards shelf contains three sections:
- Body
- Mind
- Soul

### Mobile behavior
- each shelf is horizontally scrollable
- user browses compact medals inside the shelf
- tapping one medal lifts it into foreground
- only then can the medal be flipped
- tapping outside closes directly back to the summarized shelf view

### Desktop behavior
- same concept adapted to a larger layout

---

## 14.3 Maintained achieved task in active views
If a habit is achieved and maintained:
- it continues to appear in Daily Quest
- it continues to appear in active task surfaces
- it shows a small seal marker in the top-right area of the task surface
- task detail clearly shows that it is already an achieved habit

## 15. Decision modal UX

The decision modal must feel emotionally meaningful, not administrative.

It should communicate something closer to:

**This is now an achieved habit.**

### Layout
On desktop:
- two columns

On mobile:
- stacked vertically

Each option contains:
- title
- short consequence list
- CTA

### Bullet reveal style
Reuse the same staged bullet / dot reveal behavior already used in the onboarding preparation modal.

### Option A — Maintain
Explain clearly:
- returns to Daily Quest
- stays visible in active surfaces
- becomes an achieved habit with a seal
- generates only 1 GP
- no longer recalibrates

CTA:
**Maintain habit**

### Option B — Store
Explain clearly:
- leaves active tracking
- stops generating points
- stays visible forever in the shelf
- remains saved as an achieved habit

CTA:
**Store in achievements**

### If multiple habits are achieved in the same cycle
- use one combined decision flow
- user scrolls through all achieved tasks
- decision is taken individually for each one

## 16. Entry points and notifications

There must be multiple entry points for pending achievement decisions.

## Entry point A — Rewards dot
If the user has one or more pending achievements:
- Rewards gets a dot / pulse indicator
- entering Rewards shows a prominent banner at the top
- that banner opens the decision modal flow

Concept:
**You have achieved habits to review**

This should feel similar in hierarchy to the current mode-upgrade CTA.

## Entry point B — Dashboard / task highlight
If the user enters a dashboard-related task detail route through a pending-achievement task path:
- the achievement decision modal should open first
- after decision, normal task detail behavior can continue

## Entry point C — Monthly Wrap-Up
The monthly wrap-up can also surface achieved habits.

This can be a simpler first implementation.
It is acceptable for v1 if monthly wrap-up treatment is more basic and later improved.

---

## 17. Closing behavior for the modal

If the user closes the modal without deciding:
- the state remains `achievement_pending`
- the Rewards dot remains
- the Rewards banner remains
- the pending-achievement surfaces remain active
- until either:
  - the user decides
  - or the 10-day pending window expires

---

## 18. Celebration animation after decision

After the user decides Maintain or Store:
- show a short celebratory achievement animation
- if there are multiple achieved habits, show all the seals as part of the sequence
- then land conceptually in the rewards shelf
- allow skipping with an `X`

### Multiple achievements
If there are many habits in the same cycle:
- show a combined celebration sequence with all the seals
- final landing should visually connect to the shelf

Concept:
`seal, seal, seal ... -> shelf`

---

## 19. First-time educational banner

After the user confirms achieved habits, show a contextual dashboard banner that teaches where to review them.

Concept:
- review your achieved habits in Rewards
- CTA goes directly to the rewards shelf

### Display rule
If multiple habits were achieved together:
- show one banner, not multiple banners
- no spam

## 20. Rewards integration

Rewards must be extended.

### Final direction
Extend the existing rewards/history response rather than creating a separate main endpoint.

Rewards becomes:
1. Weekly / Monthly Wrap-Up history
2. Achieved habits shelf

This is the correct product home for achieved habits.

### Payload direction for `GET /users/:id/rewards/history`
The response should be shaped around what the Rewards screen actually needs first, not around DB purity.

Recommended response shape:

```json
{
  "monthly_wrapups": [
    {
      "id": "uuid",
      "periodKey": "2026-03",
      "payload": {}
    }
  ],
  "achieved_habits": {
    "pending_count": 2,
    "body": [
      {
        "achievementId": "uuid",
        "taskId": "uuid",
        "status": "pending_decision",
        "taskName": "Walk 20 minutes",
        "pillar": "BODY",
        "traitId": "uuid",
        "traitCode": "ENERGY",
        "sealKey": "body_energy",
        "achievedAt": "2026-03-31T00:00:00.000Z",
        "gpBeforeAchievement": 50,
        "gpSinceMaintain": 0,
        "maintainEnabled": false
      }
    ],
    "mind": [],
    "soul": []
  }
}
```

### Practical rules
- `monthly_wrapups` stays compatible with the current Rewards surface.
- `achieved_habits` is shelf-ready.
- group by pillar at API level to reduce frontend transformation work.
- include `pending_count` for the Rewards dot / badge.
- each shelf item should already contain the minimum UI-ready data:
  - ids
  - status
  - task name
  - pillar
  - trait / seal identity
  - achieved date
  - GP before achievement
  - GP since maintain
  - maintain toggle state

## 22. Recommended backend services

## 22.1 `habitAchievementService`
Responsibilities:
- fetch latest 3 valid cron recalibration periods per task
- evaluate achievement rules
- create pending achievement records
- manage pending expiration
- process maintain/store decisions
- support maintained/stored toggle transitions
- expose shelf-ready data

## 22.2 Monthly hook
Integrate Habit Achievement evaluation after:
- monthly recalibration
- mode-upgrade monthly aggregation

## 22.3 Pending expiration model
`achievement_pending` should be modeled explicitly in the DB, not inferred loosely.

### Final rule
- when a task becomes pending, persist `pending_expires_at = detected_at + 10 days`
- pending state lasts until:
  - user decides, or
  - `pending_expires_at` is reached

### On expiration
When pending expires:
- remove / resolve the pending state
- return task to the normal analysis cycle
- it becomes a normal task again
- the next monthly evaluation will use the rolling current 3-month window
- the old first month naturally falls out if the window moved

### Technical direction
There are two acceptable implementation styles:
1. a daily lightweight cleanup job that resolves expired pending records
2. lazy resolution on read / on monthly run if `pending_expires_at < now`

Preferred v1 direction:
- support lazy resolution in service logic
- optionally add a cleanup job later if needed

This avoids overengineering the first version.

## 22.4 Decision service methods
Needed behavior:
- accept “maintain”
- accept “store”
- toggle stored -> maintained
- toggle maintained -> stored
- resolve pending expiration

## 24. Monthly Wrap-Up v1 direction

Monthly Wrap-Up v1 should be simple.

### Final UX direction
- same structural idea as the current Weekly Wrapped
- monthly data instead of weekly data
- plus achieved-habits section
- background gradients can change to differentiate monthly from weekly

### V1 rule
Do not invent a completely new Monthly Wrap-Up UX system for v1.
Reuse the Weekly Wrapped structure and adapt it.

This is enough for first release.

---

## 25. Final summary

Habit Achievement v1 is now defined as:

- a monthly, backend-driven, rules-based feature
- built on top of Growth Calibration history
- using a weighted 3-month performance rule
- creating a pending stand-by state before user confirmation
- excluding pending tasks from all adaptive engines and active surfaces
- forcing a user decision between:
  - **Maintain**
  - **Store**
- supporting later maintain/store toggling from the shelf
- extending Rewards with a persistent achieved-habits shelf
- preserving the achieved-habit record permanently
- treating edits after week 6 as a new task lifecycle via clone + deactivate-old behavior

This is the missing product layer that turns “real habits that last” into actual system behavior.
