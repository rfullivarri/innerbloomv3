# Admin Mode Upgrade Analysis Flow Audit

## 1) Current admin flow (exact paths/functions)

### UI trigger
- **Button**: `Run Monthly Analysis` in admin panel.
  - File: `apps/web/src/components/admin/AdminLayout.tsx`
  - Handler: `handleRunMonthlyReview`
  - Call sequence in UI:
    1. `runAdminMonthlyReview(selectedUser.id)`
    2. `fetchAdminModeUpgradeAnalysis(selectedUser.id)` to refresh displayed payload.

### Admin API entrypoint
- File: `apps/web/src/lib/adminApi.ts`
- Function: `runAdminMonthlyReview(userId)`
- Endpoint called: `POST /admin/user/:userId/run-monthly-review`

### Backend route + handler
- Route file: `apps/api/src/modules/admin/admin.routes.ts`
  - Route: `adminRouter.post('/user/:userId/run-monthly-review', postAdminRunMonthlyReview)`
- Handler file: `apps/api/src/modules/admin/admin.handlers.ts`

    1. `const now = new Date()`
    2. `await runMonthlyTaskDifficultyCalibrationForUser({ userId, now })`
    3. `const result = await runUserMonthlyModeUpgradeAggregation({ userId, now })`
    4. Return response with `period_key`, `period_start`, `next_period_start`, `processed`, `persisted`.

✅ This confirms the intended order (recalibration first, then aggregation, same `now`).

### Analysis read endpoint used by UI

- Handler: `getAdminUserModeUpgradeAnalysis` (`admin.handlers.ts`)

## 2) Root cause of original zero-result bug

The aggregation service (`runUserMonthlyModeUpgradeAggregation`) reads from `task_difficulty_recalibrations` with `r.source = 'cron'` and previous-month date bounds.

Before the fix, the admin action for mode-upgrade aggregation did **not** guarantee that monthly recalibration rows existed first for that user/month. So aggregation could legitimately process zero rows and return:
- `processed: 0`
- `persisted: 0`
- and an empty tasks analysis payload downstream.

This is consistent with the previously observed behavior.

---

## 3) What the implemented fix changed

The admin manual monthly-review route now performs both steps in one request:
1. `runMonthlyTaskDifficultyCalibrationForUser({ userId, now })`
2. `runUserMonthlyModeUpgradeAggregation({ userId, now })`

Additionally:
- both functions receive the **same `now` object**, which materially reduces period drift risk within a single request.
- user scope is explicit (`userId` passed to both calls).

---

## 4) Period consistency validation

### Aggregation period model
- In `modeUpgradeMonthlyAggregationService.ts`, default period key is previous calendar month in UTC:
  - `getPreviousMonthPeriodKey(now)`
  - `parsePeriodKey(periodKey)` => `periodStart` and `nextPeriodStart`
- Query filter:
  - `r.period_end >= periodStart`
  - `r.period_end < nextPeriodStart`

### Recalibration period model
- In `taskDifficultyCalibrationService.ts`, non-admin-run (`source='cron'`) path uses:
  - `buildAnalysisPeriod({ now, createdAt, lastPeriodEnd })`
  - period end = `endOfPreviousMonth(now)` (UTC)
- So recalibration rows inserted by this path should have `period_end` in previous month end, which aligns with aggregation filter.

### Result
- **For the current fix path**, both steps are aligned to previous calendar month and share the same `now`.
- `period_key`, `period_start`, `next_period_start` are generated by aggregation and are internally coherent.

⚠️ Subtle caveat: recalibration can produce `period_start` earlier than first day of month (because it may backfill from eligibility/lastPeriodEnd to previous month end), while aggregation groups by `period_end` month. This is not necessarily wrong, but it means "monthly" is anchored by `period_end`, not strictly `period_start`.

## 5) Source semantics (`source='cron'` for manual admin run)

Current behavior:
- `runMonthlyTaskDifficultyCalibrationForUser` forces `source: 'cron'`.
- Aggregation and analysis queries also filter `source = 'cron'`.

### Why this works
- It preserves compatibility with existing aggregation query logic.
- Manual run produces rows that are visible to aggregation immediately.

### Risks
1. **Audit ambiguity**: manual and scheduler-generated rows are indistinguishable in `task_difficulty_recalibrations`.
2. **Debugging friction**: difficult to answer "did cron run, or did an admin repair it?".
3. **Reporting contamination risk**: any cron-only dashboards lose semantic clarity.

### Safer alternative (recommended)
- Keep aggregation-compatible behavior but add provenance:
  - Option A (best): extend allowed `source` enum with `admin_monthly_review`, and change aggregation filter to `source IN ('cron', 'admin_monthly_review')`.
  - Option B (minimal schema impact): keep `source='cron'` but add metadata columns (or structured log/event) marking origin route/request id/admin id.

Given current design, this is a **traceability risk**, not an immediate correctness blocker.

## 6) Persistence and idempotency

### Re-running same user/month
- Recalibration engine checks latest `period_end` per task (`ORDER BY period_end DESC LIMIT 1`) and builds next period from `lastPeriodEnd + 1`.
- If current previous-month period already exists, it tends to skip rather than insert duplicates.

### Aggregation rerun behavior
- Aggregation transaction deletes existing rows for `(period_key, user scope)` then re-inserts from current base query.
- This is effectively idempotent at aggregate table level for same user/month.

### Remaining idempotency caveat
- There is **no DB unique constraint** on recalibrations per `(task_id, period_end, source)`.
- Concurrent runs (e.g., cron + admin at same time) could still insert duplicate monthly rows for same task.
- Aggregation uses `COUNT(*)`, so duplicates would inflate totals.

So: **functionally idempotent in serial runs, not strongly idempotent under concurrency**.

## 8) Downstream payload validation

`getUserModeUpgradeAnalysis` returns:
- `current_mode`
- `next_mode`
- `tasks_evaluated`
- `tasks_meeting_goal`
- `task_pass_rate`
- `threshold`
- `missing_tasks`
- `eligible`
- `tasks[]`

So the requested fields are present.

### Important correctness caveat in `tasks[]`
`tasks[]` query filters recalibration rows by **current user mode id now**:
```sql
AND r.game_mode_id = (SELECT game_mode_id FROM users WHERE user_id = $1::uuid LIMIT 1)
```
If user changed mode after the analyzed period, this can return empty/misaligned task rows while summary counters come from latest aggregate for current mode. This can be confusing.

### Misleading-field scenario
When there is no aggregate for current mode:
- suggestion fallback returns zero counters and `period_key = null`.
- analysis returns `tasks_evaluated=0`, `missing_tasks=0`, `eligible=false`, `tasks=[]`.

This is technically consistent but can be operationally misleading without a "no monthly aggregate found" flag.

## 9) Hidden edge cases audit

1. **User has tasks but no completions in month**
   - Recalibration rows still get inserted (completion rate 0 if expected target > 0).
   - Aggregation should include those tasks and likely mark not eligible.
   - This is correct behavior.

2. **User changed mode mid-period**
   - Recalibration currently selects latest game mode history row overall (`ORDER BY effective_at DESC LIMIT 1`) without constraining to period end.
   - This can apply the wrong weekly target for historical month analysis.
   - A helper `resolveWeeklyTarget(...)` exists but is not used in engine query path.

3. **Tasks created late in month**
   - Eligibility requires task age > 30 days at previous month end for monthly/cron path.
   - Late tasks are intentionally excluded.

4. **Archived/inactive tasks**
   - Engine reads only `t.active = TRUE`; inactive tasks are excluded.

5. **Rows already exist from cron before admin run**
   - Recalibration tends to skip due to `lastPeriodEnd`; aggregation still rebuilds from existing rows.
   - Safe in serial execution.

6. **Multiple admin runs same day**
   - Monthly review path (`source='cron'`) has no by-day dedupe but period progression logic usually prevents same-month reinserts.
   - Still subject to race condition if concurrent.

7. **Month boundary/timezone**
   - Logic uses UTC date arithmetic (`Date.UTC`, `toISOString().slice(0,10)`), reducing locale drift risk.

---

## 10) Verdict: is the fix correct, robust, safe?

### Correctness
- **Yes, largely correct** for the original issue: admin monthly review now generates monthly recalibration inputs before aggregation, with aligned `now` and user scope.

### Robustness/Safety
- **Partially robust**. Remaining risks:
  1. provenance ambiguity from `source='cron'` in manual runs,
  2. concurrency duplicate risk (no unique constraint),
  3. potential mode-history/time-travel mismatch for mid-period mode changes,
  4. possible analysis confusion when no aggregate exists (zeros without explicit reason).

---

## 11) Recommended next actions (before UI polishing / Monthly Wrap-Up)

1. **Add provenance-safe source semantics**
   - Introduce `admin_monthly_review` (or equivalent) and allow aggregation to include it.

2. **Add idempotency guard at DB level**
   - Add a unique index like `(task_id, period_end, source)` (or chosen natural key).
   - Use `INSERT ... ON CONFLICT DO NOTHING/UPDATE`.

3. **Fix historical mode selection for recalibration**
   - Select mode history row with `effective_at <= period_end`.
   - (Or use existing `resolveWeeklyTarget` flow with period-aware history retrieval.)

4. **Improve analysis response semantics**
   - Add explicit flags, e.g. `has_monthly_aggregate` and/or `analysis_period_key`, to distinguish "true zero" from "no data".

5. **Add targeted tests**
   - Concurrency/idempotency test for same user+month.
   - Mid-period mode change test ensuring historical target selection.
   - Admin monthly review provenance test (manual vs cron traceability).
